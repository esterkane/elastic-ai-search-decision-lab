# Choose a semantic search implementation

Elasticsearch provides several ways to implement semantic search. The approaches differ in how much control they give you over the inference endpoint, field configuration, and chunking behavior.

The main options are:

- **`semantic_text` workflow** (recommended): automated inference and chunking with sensible defaults
- **Inference API workflow**: manual configuration with full control over the inference endpoint
- **ELSER via ingest pipelines**: sparse vector search using a model deployed directly in Elasticsearch
- **Bring your own vectors**: use embeddings generated outside Elasticsearch

## `semantic_text` workflow (recommended) [semantic-text]

The `semantic_text` field type handles inference at ingestion time and provides sensible defaults automatically. You do not need to configure model parameters, define inference pipelines, or manage chunking manually.

Use the `semantic_text` workflow when:

- You are starting a new semantic search project and want the simplest setup
- You do not need to customize the inference endpoint configuration or chunking strategy
- You want Elasticsearch to manage model updates and chunking automatically

The `semantic_text` workflow supports dense and sparse vector models through the [inference API](/solutions/search/semantic-search/semantic-search-inference.md). For the current default inference endpoint and model, refer to [Semantic search with `semantic_text`](/solutions/search/semantic-search/semantic-search-semantic-text.md).

For an end-to-end tutorial, refer to [Semantic search with `semantic_text`](/solutions/search/semantic-search/semantic-search-semantic-text.md).

## Inference API workflow [inference-api]

The inference API workflow uses a `dense_vector` field with an inference endpoint you configure manually. This gives you explicit control over the model, similarity function, and index options.

Use the inference API workflow when:

- You need to use a specific embedding model or provider not covered by the `semantic_text` defaults
- You want to control quantization settings, similarity functions, or other index options on the vector field
- You are managing chunking manually as part of your ingest pipeline

This workflow requires more setup than `semantic_text`. You need to create an inference endpoint, define the index mapping, and build an ingest pipeline.

For an end-to-end tutorial, refer to [Semantic search with the inference API](/solutions/search/semantic-search/semantic-search-inference.md).

## ELSER via ingest pipelines [elser-ingest]

You can deploy ELSER directly in Elasticsearch and use it through an ingest pipeline with a `sparse_vector` field, without using the inference API. This is the most manual workflow for sparse vector search.

Use ELSER via ingest pipelines when:

- You are on an Elasticsearch version before inference API support was available
- You need full control over the ingest pipeline configuration
- You require explainability: ELSER produces token-weight expansions that are easier to inspect than dense embeddings

This approach requires deploying the ELSER model through the Eland client, creating an index mapping with a `sparse_vector` field, and configuring an ingest pipeline to run inference at ingestion time.

For an end-to-end tutorial, refer to [Semantic search with ELSER (ingest pipelines)](/solutions/search/semantic-search/semantic-search-elser-ingest-pipelines.md).

## Bring your own vectors [byov]

If you generate embeddings outside Elasticsearch — using an external model or service — you can index them directly into a `dense_vector` field without using the inference API.

Use this approach when:

- Embeddings are generated by an external pipeline or service that Elasticsearch does not need to call
- You are working with a proprietary or fine-tuned model that is not available through the inference API
- You want full control over the embedding generation process

For more information, refer to [Bring your own dense vectors](/solutions/search/vector/bring-own-vectors.md).

## Comparison [comparison]

| | `semantic_text` | Inference API | ELSER ingest pipeline | Bring your own vectors |
|---|---|---|---|---|
| Setup complexity | Low | Medium | High | Medium |
| Model flexibility | Inference API services | Inference API services | ELSER only | Any external model |
| Chunking control | Automatic | Manual | Manual | Manual |
| Sparse vector support | Yes (ELSER) | Yes | Yes | No |
| Recommended starting point | ✓ | | | |
